# MCU Map 文件分析网站设计文档

## 项目概述

开发一个基于 Web 的 MCU map 文件分析工具，支持上传 GCC 编译器生成的 map 文件，并通过图形化界面展示内存使用情况、函数大小排名、模块统计等关键信息，帮助开发者优化嵌入式程序的内存占用。

## 设计目标

- 支持 GCC 编译器生成的 map 文件格式
- 提供全面的内存分析视图（Flash/RAM 使用、段分布、模块统计）
- 仪表板风格的用户界面，多图表同时展示
- 纯前端实现，无需后端服务器
- 不保存用户数据，所有分析在浏览器本地完成

## 技术方案

### 选定方案：纯前端方案

**技术栈：**
- React 18 + TypeScript
- Vite（构建工具）
- ECharts（图表库）
- Tailwind CSS（样式框架）
- React hooks（状态管理）

**方案优势：**
- 部署简单，只需静态托管（GitHub Pages、Vercel、Netlify）
- 无需后端服务器，零运维成本
- 数据完全在本地处理，隐私性好
- 响应速度快，无网络延迟

4. 转换为图表数据格式
5. 渲染多个可视化图表

## 数据模型

### 解析后的数据结构
```typescript
interface ParsedMapData {
  memoryRegions: {
    name: string;        // 如 "FLASH", "RAM"
    origin: number;      // 起始地址
    length: number;      // 总大小
    used: number;        // 已使用
    free: number;        // 剩余
  }[];

  symbols: {
    name: string;        // 符号名称
    address: number;     // 地址
    size: number;        // 大小
    section: string;     // 所属段（.text, .data, .bss等）
    file: string;        // 来源文件
  }[];

  modules: {
    name: string;        // 模块/文件名
    totalSize: number;   // 总大小
    textSize: number;    // 代码段大小
    dataSize: number;    // 数据段大小
    bssSize: number;     // BSS段大小
  }[];
}
```

### 解析流程
1. 使用正则表达式匹配各个段落
2. 提取内存区域信息（Memory Configuration 段）
3. 解析符号表（从 .text、.data、.bss 等段中提取）
4. 按文件聚合统计（从符号的来源文件分组）
5. 计算使用率和排名

## 用户界面设计

### 页面布局

**1. 顶部区域**
- 应用标题和简短说明
- 文件上传区（支持拖拽和点击选择）
- 文件信息显示（文件名、大小、解析状态）

**2. 总览卡片区**（6个卡片，2行3列）
- Flash/ROM 使用率（百分比 + 进度条 + 绝对值）
- RAM 使用率（百分比 + 进度条 + 绝对值）
- 总符号数量
- 最大函数大小
- 对齐填充浪费空间
- 潜在未使用符号数量

**3. 主要图表区**（分标签页组织）

**标签页 1：内存概览**
- 内存分布饼图（Flash vs RAM）
- 内存段分布堆叠柱状图（.text、.data、.bss、.rodata 等）
- 堆栈分配可视化

**标签页 2：模块分析**
- 模块内存占用树状图（Treemap，颜色深浅表示大小）
- Top 20 模块大小排名柱状图
- 静态库占用分析（饼图或柱状图）

**标签页 3：函数分析**
- Top 20 最大函数柱状图（可交互）
- 函数大小分布直方图（了解函数大小分布特征）

**标签页 4：优化建议**
- 对齐填充分析表格（显示哪些符号有对齐浪费）
- 潜在未使用符号列表（需要进一步验证）
- 重复定义符号警告
- 内存碎片可视化（地址空间图）

**4. 详细表格区**（可折叠，支持多种视图切换）
- 所有符号列表（可排序、可搜索、分页显示）
- 列：符号名、大小、地址、段、来源文件/模块、对齐信息
- 对象文件列表（按大小排序）
- 导出功能（CSV、JSON）

### 交互特性
- 图表之间联动（点击模块，其他图表高亮相关数据）
- 支持导出分析报告（JSON/CSV 格式）
- 响应式设计，支持移动端查看
- 可以设置阈值（如"显示大于 1KB 的函数"）

## 分析维度

### 1. 内存布局
- ROM/Flash 使用量 — 代码段、只读数据占用情况
- RAM 使用量 — 数据段、BSS 段、堆栈分配
- 各段分布 — .text、.data、.bss、.rodata 等

### 2. 模块/函数级别
- 各模块大小排名 — 找出占用大的模块
- 函数大小排名 — 识别大函数，优化候选
- 符号地址 — 定位具体函数/变量位置

### 3. 对象文件分析
- 静态库占用 — 哪些库占用了多少空间
- 潜在未使用符号 — 链接时可能未移除的死代码
- 重复定义 — 多次定义的符号

### 4. 对齐与碎片
- 对齐填充 — 编译器对齐造成的空洞
- 内存碎片 — 不连续的内存布局

## 错误处理与边界情况

### 文件验证
- 检查文件扩展名（.map）
- 验证文件大小（建议限制在 50MB 以内，超过给出警告）
- 检查文件格式是否为 GCC map 文件（通过特征字符串识别）

### 解析错误处理
- 如果无法识别 map 文件格式，显示友好的错误提示
- 部分解析失败时，显示已解析的内容 + 警告信息
- 提供示例 map 文件下载链接，帮助用户理解支持的格式

### 性能优化
- 大文件解析时显示进度条
- 使用虚拟滚动渲染大型表格（避免一次性渲染数千行）
- 图表数据量过大时自动采样（如只显示 Top 100）

### 用户体验细节
- 首次访问显示引导提示
- 支持键盘快捷键（如 Ctrl+O 打开文件）
- 提供"清除数据"按钮重新开始
- 暗色模式支持

## 技术实现细节

### 核心解析算法
```typescript
// 解析策略
1. 按行读取 map 文件
2. 使用状态机识别不同段落：
   - Memory Configuration
   - Linker script and memory map
   - Cross Reference Table
3. 针对每个段落使用特定的正则表达式提取数据
4. 计算派生指标（使用率、排名、对齐浪费等）
```

### 关键正则表达式模式
- 内存区域：`/^(\w+)\s+0x([0-9a-f]+)\s+0x([0-9a-f]+)/`
- 符号定义：`/^\s+0x([0-9a-f]+)\s+(\w+)\s+0x([0-9a-f]+)\s+(.+)$/`
- 段信息：`/^\.(\w+)\s+0x([0-9a-f]+)\s+0x([0-9a-f]+)/`

### 对齐分析算法
- 检测连续符号之间的地址间隙
- 如果间隙小于 16 字节且不是 2 的幂次，标记为可疑对齐
- 统计总对齐浪费空间

### 未使用符号检测（启发式）
- 标记没有被其他符号引用的符号
- 排除入口点和中断向量表
- 需要用户手动验证（因为可能是动态调用）
